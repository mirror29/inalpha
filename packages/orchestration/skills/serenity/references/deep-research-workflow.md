# 深度主题扫描工作流

用户要"当下值得研究什么 / 给我排个优先级 / 完整扫一遍这个主题"时用本文档。数据纪律以 SKILL.md 的映射表为准：每个"查"的步骤都是一次工具调用。

## 完成标准

宽主题的最终回答之前尽量做到：
- 覆盖 ≥3 个产业链层级，且**层级排序先于公司排序**
- 候选池 ≥20 家再筛（市场上市公司足够多时）；信息源 ≥25 条（工具与轮次允许时）
- 每个最终候选有具体证据 + 失效条件
- 至少点名一个热门 / 显眼方向被降级及原因
- 给出用户下一步该跑的核实动作

工具或轮次不够时：明确声明这是初步结论，列出还差哪些来源核实，不要装完整。

## 步骤

### 1. 定范围
市场 / 主题 / 时间窗（"现在"默认 3-12 个月）。缺的范围会实质改变答案时才追问，否则合理推断。

### 2. 叙事 → 系统变化
写出链条：`需求浪潮 → 系统压力 → 必需的技术变化 → 被约束的层级`。
找最关键的物理约束：功耗、延迟、带宽、散热、良率、纯度、可靠性、周期时间、封装密度、监管、电网接入。

### 3. 产业链分层清单
1. 终端客户与资本开支来源
2. 系统集成 / OEM
3. 模组与子系统
4. 芯片、器件、关键部件
5. 工艺、组装、封装、测试
6. 设备与量测
7. 材料、耗材、特种投入
8. 物理基础设施

分桶保持颗粒度：经济学不同就拆开（算力芯片 / EDA·IP / 存储互连 / 设备 / 材料 / 测试 / 封装 / 光链路 / PCB·CCL / 供电散热各算各的）。

### 4. 找稀缺层
信号叠加越多越值得看：客户绕不开、供应商数量少、认证慢、扩产需要专用设备 / 许可 / know-how / 材料纯度、客户用预付款 / 锁产能 / 加急单 / 接受涨价表达紧迫、市场还按旧业务类别给它分类。
证据支持时优先看不那么显眼的上游层。**此步结束先写出层级排序，再进公司名单。**

### 5. 建候选池
铺开后再排，不要从热门票出发：显眼龙头、上游设备、专用工艺工具、材料耗材、测试量测、先进封装、基础设施与供电，以及需要被显式降级的热门名字。
每家平话归类：控制稀缺层 / 供给稀缺层 / 受益但控制力弱 / 有敞口没定价权 / 只有故事。

### 6. 收集当期证据
来源优先级与分级见 `evidence-ladder.md`；按市场选信息源路径见 `market-source-playbook.md`。
执行手段：`web.search_news` / `web.search` 扫公告与新闻线索；**关键来源用 `web.fetch` 读正文**（snippet 只能当线索，正文 + 发布日期才够给证据定级）；候选公司代码用 `data.search_symbol` 解析后再 `data.get_fundamentals` 验财务；`data.get_bars`（默认 fresh）看近期走势是否已计入；`factor.timing` 做技术面交叉验证；复杂主题可一次 `research.deep_dive` 拿多分析师视角。

### 7. 排序
按 SKILL.md 的 rubric 打分。两套排序分开呈现：层级排序（系统里哪些部分先看）、公司排序（哪些公司最能代表这些层级且有证据）。
每个最终候选答全：卡住的环节 / 链上位置 / 排序原因 / 证据 / 什么会让排名变弱。

### 8. 输出
结论先行：优先层级 → 优先名单 → 排序理由 → 最强证据 → 被降级的热门方向 → 判断最容易错在哪 → 下一步查什么。
散文为主，表格只用于排序或证据对比。**结尾标注数据截止时间。**
